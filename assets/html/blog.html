From 0% to 95% flaky test visibility — without a paid observability platform

Flaky tests silently erode CI trust. I built a cross-run history tracker using GitHub Actions artifacts as free key-value storage — no database, no external service. Here's exactly how it works.

Read the case study

The problem

The Cypress E2E suite had 7 spec files and ~20 tests. On any given push, 2–3 tests would fail seemingly at random, then pass on a re-run. Engineers stopped trusting the pipeline. When CI is a coin flip, teams either add retry logic that masks real failures, or they ignore failures entirely. Both outcomes are worse than having no CI at all.

The root cause was multi-layered: some tests made live Lambda API calls with no timeout buffer, some depended on CSS selectors that changed with deployments, and one test was asserting on text that CSS text-transform: uppercase was silently transforming at render time.

The constraint

No paid observability tools (DataDog, Sentry, TestRail). Everything had to run on the GitHub Actions free tier with zero external services.

The solution

I built a Flaky Test Detector agent that runs after every Cypress CI execution. The key insight: GitHub Actions artifact storage is effectively a free, persistent key-value store. Each run:

  1. Downloads the latest flaky-history artifact (a JSON file tracking every test's pass/fail across the last 30 runs)
  2. Merges the current run's Mochawesome JSON results into the history
  3. Computes a flip rate per test — the fraction of consecutive run-pairs where the outcome changed
  4. Flags tests with a flip rate ≥ 20% and posts a PR comment with a sparkline: 🟩🟥🟩🟥🟩
  5. Re-uploads the updated history as overwrite: true for the next run to consume

The Lambda timeout tests were fixed with cy.intercept() stubs. The CSS text-assertion was fixed with Should Contain ... ignore_case=${True} in Robot Framework and a case-insensitive check in Cypress. The selector brittleness was fixed by migrating to data-testid attributes.

Result

After 10 runs the baseline stabilized. Flaky test rate: 0%. Pipeline trust restored. Total cost: $0. The agent itself runs in under 15 seconds.

Blocking bad merges before review — a PR quality gate with zero paid tooling

SonarQube costs money. But the three things it actually enforces — coverage thresholds, test debt, and commit hygiene — can be replicated in 200 lines of Python. Here's the implementation.

Read the case study

The problem

PRs were merging with skipped tests (.skip, xit), dropping coverage, and non-descriptive commit messages like "fix" or "update". These are small things individually, but they compound — a codebase with 40 skipped tests and no coverage enforcement is a codebase you can't refactor safely.

The approach

A single Python script runs on every pull_request event and performs three static checks:

  1. Coverage — parses coverage/lcov.info (generated by Jest) and compares lines/functions/branches against configurable thresholds
  2. Test debt — scans all .cy.js, .test.js, .spec.ts, .robot, and .feature files for skip/only/ignore markers across all 5 frameworks
  3. Conventional commits — fetches the PR's commit list via the GitHub API and validates each against the Conventional Commits 1.0 regex

The result is posted as a single PR comment that updates in place on each push (using an HTML marker to find and PATCH the existing comment) — no comment spam.

The key design decision

Regressions are hard-blocked. New .only tests are a hard block (they silently filter your entire test suite in CI, producing false-green builds). Coverage drops are configurable. Commit messages are hard-blocked on master but can be bypassed with a dispatch input for hotfixes.

An AI agent that reads Cypress failures, classifies root causes, and opens auto-fix PRs

Using Claude's tool-use API to build a fully agentic CI failure analyst — one that reads test logs, fetches source files, classifies errors, and opens a GitHub PR with the patch. No human in the loop for deterministic fixes.

Read the case study

The premise

Most CI failures have deterministic root causes: a stale selector, a wrong assertion value, a missing .should() chain. These don't require human judgment to fix — they require pattern recognition and a git commit. That's an agent's job.

The architecture

The agent runs as a Python script inside a GitHub Actions runner. When a Cypress run fails, it:

  1. Parses the Mochawesome JSON to extract all AssertionError and CypressError failures
  2. For each failure, runs an agentic Claude loop with three tools: read_test_file (fetches the spec from GitHub API), classify_failure (determines error type, root cause, patchability, confidence), and generate_patch (produces exact string-replace diff)
  3. Bundles all generated patches into a single branch and opens one auto-fix PR
  4. Posts a rich comment on the triggering PR with the classification and patch diff

The safety model

The agent only sets patchable=true for deterministic fixes: stale CSS selectors, wrong assertion values, typos in expected text. It explicitly refuses to patch timing issues, network dependencies, and environment differences — those require human investigation.

What this means for QA consulting

In a client engagement, this pattern means junior engineers spend less time triaging "why did CI fail again?" and more time writing tests. The agent handles the mechanical debugging loop. Senior engineers review the auto-fix PR in 30 seconds instead of spending 20 minutes on root cause analysis.

Building a full testing pyramid for a static portfolio site — why and how

A portfolio site doesn't "need" mutation testing or contract tests. But building the full pyramid was the point — proving I know when and how to apply each layer, not just that I can write Cypress tests.

Read the case study

The intentional over-engineering

This site is a Jekyll static site with 8 HTML pages, a contact form backed by Lambda, and no database. A reasonable QA strategy would be: Cypress for E2E, done. Instead I built all seven layers of the testing pyramid. Here's why each one exists and what it proves.

  • Jest unit tests — validates the pure business logic functions (form validation, data transforms) in isolation. Caught 2 edge cases that Cypress would have missed.
  • Pact contract tests — verifies the Lambda API's request/response schema hasn't silently changed between deployments. The consumer test runs client-side; the provider verification runs against the live Lambda.
  • StrykerJS mutation tests — kills mutants in the unit test suite to verify the tests are actually detecting logic errors (not just running and passing).
  • Karate API tests — health checks and payload validation for the Lambda endpoints, independent of the UI. Catches infrastructure issues that Cypress can't distinguish from test failures.
  • Robot Framework BDD — 6 suites written in Given/When/Then style. Demonstrates the ability to write tests that non-technical stakeholders can read and validate.
  • Playwright cross-browser — runs the full E2E suite against Chromium, Firefox, WebKit, and mobile Chrome. Caught 2 CSS layout issues that only appeared in Firefox.
  • Lighthouse CI — enforces performance, accessibility, and SEO thresholds. Caught a missing aria-label and a font-loading performance regression.

The real lesson

The value isn't having all 7 layers on a portfolio site. The value is knowing what each layer catches that the others don't — and being able to have that conversation with a client who wants to know where to invest their testing budget.
